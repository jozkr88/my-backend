# Joz LLM Refactor Plan

This document is the working handoff for cleaning up the Joz LLM architecture.

It is written so another Codex session or account can continue the work without
reconstructing the intent from chat history.

## Goal

Build a clean architecture where:

- authoring is separate from runtime retrieval
- frontend is presentation-only
- backend owns ranking, recruiter logic, and lane behavior
- larger brand/user-scale proof ranks above smaller current-project mentions
- each lane can be improved with the same repeatable workflow

## Problem statement

The system has been drifting because several responsibilities are mixed:

- source authoring
- normalized record generation
- runtime retrieval
- ranking policy
- recruiter answer policy
- frontend label config

This causes repeated rounds of manual tuning and makes it hard to see what the
real source of truth is.

## Target architecture

### 1. Authoring layer

Source of truth for human editing:

- `data/joz/inbox/*.md`
- `data/joz/inbox/*.meta.json`

Each source record should include:

- `slug`
- `title`
- `category`
- `lane`
- `summary`
- `tags`
- `impact_score`
- `priority_label`
- `verification`
- `claims`
- `proof_points`
- `regions`
- `companies`

Recommended additional fields when useful:

- `years_experience`
- `years_ai_ml`
- `nationality`
- `work_authorization`
- `location`
- `availability`
- `contact_channels`

These are especially useful for recruiter-facing records.

### 2. Processing layer

Generated by ingestion:

- `server/tools/build-joz-knowledge.js`

Responsibilities:

- validate metadata
- normalize source records
- preserve verification state
- carry ranking metadata such as `impact_score` and `priority_label`
- produce deterministic published outputs

### 3. Runtime knowledge layer

Backend runtime should retrieve from:

- `joz_documents`
- `joz_document_chunks`
- retrieval metadata

At the moment the file-based bundle is acceptable as an iteration step, but the
long-term serving source of truth should be a database-backed store.

### 4. Retrieval layer

Owned by backend only.

Current retrieval entry point:

- `server/db.js`

Ranking should prefer:

1. correct lane
2. verified material
3. high `impact_score`
4. user scale / enterprise complexity
5. strong brand recognition
6. measurable outcomes
7. recency only after the above

### 5. Prompt / answer policy layer

Keep this explicit and compact.

Current prompt and fallback behavior live in:

- `server/shared/jozLlmProfile.js`

This file currently does too much. Long-term it should likely be split into:

- `server/knowledge/policies/recruiterAnswers.js`
- `server/knowledge/policies/systemPrompt.js`
- `server/knowledge/policies/fallbacks.js`

### 6. Frontend layer

Frontend should only contain:

- lane labels
- UI summaries
- button labels
- display logic

Frontend should not contain:

- raw Joz knowledge corpus
- retrieval heuristics
- private recruiter logic
- hidden JSON pretending to be private

## Lane-by-lane workflow

Use this same sequence for every lane.

### Step 1. Define lane intent

Clarify:

- what this lane is for
- what the user is trying to learn
- what proof should surface first
- what should never dominate the lane

### Step 2. Audit current lane data

Inspect:

- `server/shared/jozLlmLanes.js`
- `server/shared/jozLlmProfile.js`
- all `data/joz/inbox/*.meta.json` where `lane` matches
- generated `normalized` / `published` files

### Step 3. Create lane-specific proof records

Do not rely on one giant CV file.

Instead create:

- hero proof record
- recruiter FAQ if needed
- lane-specific business or mindset record
- scale-priority proof record if ranking matters
- contact / logistics record if recruiters will ask operational questions

### Step 4. Add ranking metadata

Each important record should get:

- `impact_score`
- `priority_label`

Suggested `priority_label` values:

- `hero`
- `high`
- `standard`
- `supporting`

### Step 5. Rebuild knowledge

Run:

```bash
cd server
npm run build:joz-knowledge
```

Then inspect:

- `data/joz/normalized/*.json`
- `data/joz/published/joz-documents.generated.json`
- `data/joz/published/joz-knowledge-report.json`

### Step 6. Verify runtime behavior

Check:

- retrieval order
- recruiter-style wording
- lane action labels
- whether the biggest proof is surfacing first

### Step 7. Push only scoped backend changes

When syncing to `jozkr88/my-backend`, push only:

- changed backend logic
- changed `data/joz` source files
- changed generated `normalized` / `published` files

Do not mix unrelated frontend noise or local build junk into the commit.

## Supabase decision

Supabase is not required immediately.

Recommended rule:

- use file-based authoring while the content model is still stabilizing
- move runtime knowledge to Supabase once:
  - privacy matters
  - the corpus grows
  - chunk retrieval is needed
  - nontechnical editing matters
  - ranking and provenance need cleaner operational structure

Long-term best practice:

- files = authoring input
- Supabase = runtime serving source of truth

## Privacy rule

Anything shipped in a frontend bundle is public.

Anything committed to a public repo is public.

Therefore:

- Joz knowledge corpus should remain backend-side
- frontend should only get the minimum response needed for display
