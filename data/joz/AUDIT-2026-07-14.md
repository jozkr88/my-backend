# Joz Dataset Audit 2026-07-14

## Scope

This audit covers the current Joz knowledge dataset under `data/joz` and the runtime retrieval path that consumes the published output.

## What Exists

- Dataset type: knowledge corpus, not ERP/finance/HR operational data
- Raw source layer: `data/joz/inbox`
- Structured record layer: `data/joz/normalized`
- Shared taxonomy layer: `data/joz/ontology`
- Aggregate publish layer: `data/joz/published`

Current published counts at audit time:

- `60` source files
- `60` normalized records
- `18` verified records
- `81` ontology entities
- `12` proof objects

Lane coverage:

- `37` `skills`
- `11` `business_need`
- `10` `systems_mindset`
- `2` `booking`

## Strengths

- Every source record has matching sidecar metadata.
- The builder validates categories, lanes, ontology references, and required metadata.
- The ontology already captures business-relevant dimensions:
  - `problems`
  - `principles`
  - `capabilities`
  - `outcomes`
  - `governance`
  - `industries`
  - `proofs`
- Stronger records already include:
  - claims
  - proof points
  - measurable outcome counts
  - enterprise scale score
  - related proofs

## Gaps Found

### 1. Verification policy was looser in runtime than in workflow

The workflow said only verified material should be strong publishable evidence, but the published aggregate exposed the entire corpus to retrieval, including unresolved content.

### 2. Verification statuses are useful but not clearly tiered

The dataset uses more than:

- `draft`
- `needs_review`
- `verified`

It also uses:

- `cv_supported`
- `project_supported`
- `capability_supported`
- `positioning_supported`
- `framework_supported`
- `cv_and_project_supported`
- `user_provided`

This is workable, but the runtime needs a clear distinction between:

- strong evidence
- usable support
- unresolved material

### 3. Some mindset/system records are structurally thin

Several records have ontology tags but no explicit `claims` or `proof_points`. These are useful as philosophical framing, but weak as evidence-backed retrieval units.

### 4. The dataset is still identity-heavy

It is strong on:

- who Joz is
- proof of experience
- recruiter answers
- business value framing
- systems reasoning

It is not yet strong on:

- profit diagnostic frameworks by function
- detailed operating-model use cases
- department-specific transformation patterns
- explicit process maps across finance, ERP, HR, marketing, ops, and growth

## Changes Implemented

### Runtime-safe publish split

The build now publishes two distinct sets:

- `records`
  Full normalized corpus for audit and inspection
- `model_ready_records`
  Runtime-safe subset excluding `draft` and `needs_review`

This preserves auditability without letting unresolved material enter the default retrieval path.

### Retrieval now prefers model-ready records

The server now loads `model_ready_records` when available and falls back to filtering compatible statuses if the new field is absent.

### Status policy clarified

The README now distinguishes:

- `verified` as strong evidence
- supported statuses as secondary retrieval context
- `draft` and `needs_review` as non-runtime material

## Next Extension Plan

### Tier 1

- Add explicit `claims` and `proof_points` to thin `systems_mindset` records.
- Review and either verify or isolate unresolved company-context research.
- Add `intent_families` and `sub_intents` to high-value business records so routing becomes more precise.

### Tier 2

Extend the corpus from identity/proof into business-operating intelligence:

- `finance`
  - margin improvement
  - spend control
  - working capital
  - close acceleration
- `ERP and operations`
  - exception handling
  - workflow bottlenecks
  - procurement and inventory decision support
- `HR`
  - onboarding
  - manager support
  - recruiting evaluation
  - capability scaling
- `marketing and growth`
  - campaign effectiveness
  - conversion diagnostics
  - CAC efficiency
  - customer journey friction
- `leadership`
  - operating model redesign
  - AI adoption governance
  - KPI-linked transformation sequencing

### Tier 3

If the long-term goal is true profit and operating intelligence, add separate structured business datasets instead of forcing everything into the content corpus:

- opportunity and revenue datasets
- cost and process datasets
- workflow telemetry datasets
- approval and exception datasets
- outcome-labeled decision datasets

## Priority Recommendation

The immediate next move should be to enrich the existing Joz dataset with more function-specific business records before introducing a separate operational data layer.

That sequence is lower risk because it improves retrieval quality, message precision, and proof-based business positioning without requiring ERP-grade integration first.
